How to design software:

  1. obtain requirements
  2. draw use case diagram (Draw the actors, and draw a line to what they need from the program; what they need are the goals of the program)
  3. create folders for groups
  4. choose one goal prioritize the one that is
    1. most important
    2. most difficult
    3. most unknown
  5. Write a use case for that goal:
    1. clear value step
    2. start and stop
    3. external initiator
    4. happy path and alternative paths
  6. Circle all nouns and consider each as a potential class.
  7. Draw UML of classes. All verbs are potential methods to add.
  8. Add getters and setters and relationships
  9. Code
  10. Ask self:
    1. Does this satisfy the customer or goal?
    2. Are OOP used?
    3. Flexible for change?
  11. Repeat 4 to 10 until all goals are complete

How to do a data project:

  1. Choose a datastore.
  2. Model the data with a diagram
  3. Build the model in your program (Remove redundancy, PK, FK, indexes, proper data types)
  4. Find how to create test data for unit tests later
  5. How to load data into your model
  6. Automate producers for the models
  7. Build consumers for the models (Including aggregation functions) (Optional UI)
  8. Automate unit tests using the test data
  9. Data Integrity Scripts and alerts
  10. Performance tuning
  11. Monitor Scheduled ETL Jobs
  12. Documentation in README (Title, executive summary, example, features, installation, support, FAQ)

Using MySQL Instead of NoSQL With Lots of Data

Based on advice from the Wix Engineering site.

  • Use simple queries
  • Keep redundant data
  • Fields are either text blobs or indexed
  • Don't use transactions
